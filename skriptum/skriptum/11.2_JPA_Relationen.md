# 11.2 JPA – Relationen zwischen Entitäten

Nachdem du grundlegende JPA-Operationen kennst, lernst du jetzt, wie du **Beziehungen zwischen Tabellen** mit JPA abbildest – das Herzstück von ORM.

## Datenbankschema

Für dieses Kapitel nehmen wir folgendes Schema an:

```sql
CREATE TABLE patient (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL
);

CREATE TABLE adresse (
    id INT AUTO_INCREMENT PRIMARY KEY,
    strasse VARCHAR(100),
    patient_id INT UNIQUE,
    FOREIGN KEY (patient_id) REFERENCES patient(id)
);

CREATE TABLE befund (
    id INT AUTO_INCREMENT PRIMARY KEY,
    diagnose VARCHAR(200),
    patient_id INT,
    FOREIGN KEY (patient_id) REFERENCES patient(id)
);
```

## Arten von Relationen

| Relation | Java-Annotation | DB-Umsetzung |
|----------|----------------|--------------|
| 1:1 | `@OneToOne` | Fremdschlüssel (UNIQUE) |
| 1:N | `@OneToMany` / `@ManyToOne` | Fremdschlüssel auf der N-Seite |
| M:N | `@ManyToMany` | Kreuztabelle |

## @ManyToOne und @OneToMany

Der häufigste Fall: Ein Patient hat viele Befunde, ein Befund gehört zu genau einem Patienten.

### Die owning Seite: @ManyToOne (Befund → Patient)

```java
@Entity
@Table(name = "befund")
public class Befund {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int id;

    private String diagnose;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "patient_id")
    private Patient patient;

    // Konstruktor, Getter, Setter
}
```

| Annotation | Bedeutung |
|-----------|-----------|
| `@ManyToOne` | Viele Befunde können zu einem Patienten gehören |
| `@JoinColumn(name = "patient_id")` | Fremdschlüssel-Spalte in der `befund`-Tabelle |
| `fetch = FetchType.LAZY` | Patient wird erst bei Zugriff geladen (siehe unten) |

### Die inverse Seite: @OneToMany (Patient → Befunde)

```java
@Entity
@Table(name = "patient")
public class Patient {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int id;

    private String name;

    @OneToMany(mappedBy = "patient", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Befund> befunde = new ArrayList<>();

    // Konstruktor, Getter, Setter
}
```

| Annotation | Bedeutung |
|-----------|-----------|
| `mappedBy = "patient"` | Die andere Seite (`Befund.patient`) besitzt die Beziehung |
| `cascade = CascadeType.ALL` | Operationen auf Patient werden auf Befunde übertragen |
| `orphanRemoval = true` | Entfernte Befunde werden automatisch gelöscht |

**Wichtig:** Bei `@OneToMany` mit `mappedBy` wird die Beziehung von der `@ManyToOne`-Seite verwaltet. Du musst die Befunde manuell zum Patienten hinzufügen:

```java
Patient p = new Patient("Anna Müller");
Befund b = new Befund("Hypertonie");
b.setPatient(p);           // Beziehung setzen (wichtig!)
p.getBefunde().add(b);     // Referenz auch auf der anderen Seite

em.persist(p);             // Patient + Befund werden gespeichert
```

## @OneToOne

Jeder Patient hat genau eine Adresse, jede Adresse gehört zu genau einem Patienten.

```java
@Entity
@Table(name = "adresse")
public class Adresse {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int id;

    private String strasse;

    @OneToOne
    @JoinColumn(name = "patient_id", unique = true)
    private Patient patient;
}
```

```java
@Entity
@Table(name = "patient")
public class Patient {

    @OneToOne(mappedBy = "patient", cascade = CascadeType.ALL)
    private Adresse adresse;
}
```

## @ManyToMany

Ein Patient kann mehrere Medikamente bekommen, ein Medikament kann mehreren Patienten verschrieben werden. Das erfordert eine Kreuztabelle.

```sql
CREATE TABLE medikament (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL
);

CREATE TABLE patient_medikament (
    patient_id INT NOT NULL,
    medikament_id INT NOT NULL,
    PRIMARY KEY (patient_id, medikament_id),
    FOREIGN KEY (patient_id) REFERENCES patient(id),
    FOREIGN KEY (medikament_id) REFERENCES medikament(id)
);
```

```java
@Entity
@Table(name = "medikament")
public class Medikament {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int id;

    private String name;

    @ManyToMany(mappedBy = "medikamente")
    private List<Patient> patienten = new ArrayList<>();
}
```

```java
@Entity
@Table(name = "patient")
public class Patient {

    @ManyToMany(cascade = {CascadeType.PERSIST, CascadeType.MERGE})
    @JoinTable(
        name = "patient_medikament",
        joinColumns = @JoinColumn(name = "patient_id"),
        inverseJoinColumns = @JoinColumn(name = "medikament_id")
    )
    private List<Medikament> medikamente = new ArrayList<>();
}
```

## Lazy vs. Eager Fetching

Beim Laden einer Entity entscheidet die Fetch-Strategie, ob abhängige Daten sofort oder erst bei Bedarf geladen werden:

| Fetch-Typ | Bedeutung |
|-----------|-----------|
| `FetchType.LAZY` | Wird erst bei Zugriff geladen (Proxy) – spart Ressourcen |
| `FetchType.EAGER` | Wird sofort mit einer JOIN-Abfrage geladen |

Standardwerte:

| Annotation | Default |
|-----------|---------|
| `@ManyToOne` | `EAGER` |
| `@OneToMany` | `LAZY` |
| `@OneToOne` | `EAGER` |
| `@ManyToMany` | `LAZY` |

Empfehlung: Verwende `LAZY` wo möglich, um unnötige Datenbankabfragen zu vermeiden.

```java
@Entity
public class Befund {
    @ManyToOne(fetch = FetchType.LAZY)   // explizit lazy
    private Patient patient;
}
```

### LazyInitializationException

Greifst du auf eine lazy-Assoziation außerhalb einer offenen Transaktion zu, erhältst du eine `LazyInitializationException`:

```java
em.getTransaction().begin();
Patient p = em.find(Patient.class, 1);
em.getTransaction().commit();

System.out.println(p.getBefunde().size());  // LazyInitializationException!
```

Lösungen:
- Daten innerhalb der Transaktion laden (z. B. durch Aufruf von `p.getBefunde().size()`)
- `FetchType.EAGER` verwenden (nicht empfohlen bei großen Datenmengen)
- Mit `JOIN FETCH` in JPQL explizit laden (empfohlen):

```java
TypedQuery<Patient> query = em.createQuery(
    "SELECT p FROM Patient p JOIN FETCH p.befunde WHERE p.id = :id", Patient.class);
query.setParameter("id", 1);
Patient p = query.getSingleResult();
// p.getBefunde() ist jetzt gefüllt
```

## Cascading

Cascading bestimmt, welche Operationen von einer Eltern-Entity auf abhängige Entities übertragen werden:

```java
@OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
private List<Befund> befunde;
```

### CascadeType-Optionen

| Typ | Überträgt |
|-----|-----------|
| `PERSIST` | `em.persist()` auf Kind-Elemente |
| `MERGE` | `em.merge()` auf Kind-Elemente |
| `REMOVE` | `em.remove()` auf Kind-Elemente |
| `REFRESH` | `em.refresh()` auf Kind-Elemente |
| `DETACH` | `em.detach()` auf Kind-Elemente |
| `ALL` | Alle oben genannten |

### Beispiele

**Cascade = ALL** (speichern + löschen):

```java
Patient p = new Patient("Anna");
Befund b = new Befund("Hypertonie");
p.getBefunde().add(b);
b.setPatient(p);

em.persist(p);  // Patient + Befund werden gespeichert (Cascade.PERSIST)
em.remove(p);   // Patient + Befund werden gelöscht (Cascade.REMOVE)
```

**Cascade = PERSIST + MERGE** (nur speichern, nicht löschen):

```java
@OneToMany(cascade = {CascadeType.PERSIST, CascadeType.MERGE})
private List<Befund> befunde;
```

### orphanRemoval

`orphanRemoval = true` löscht automatisch Kind-Elemente, die aus der Collection entfernt werden:

```java
p.getBefunde().remove(0);  // Befund wird beim nächsten commit aus der DB gelöscht
```

## Übersicht: Relationen konfigurieren

| Beziehung | Owning Seite | Inverse Seite |
|-----------|-------------|---------------|
| 1:1 | `@OneToOne` + `@JoinColumn` | `@OneToOne(mappedBy = "feld")` |
| 1:N | `@ManyToOne` + `@JoinColumn` (N-Seite) | `@OneToMany(mappedBy = "feld")` |
| M:N | `@ManyToMany` + `@JoinTable` | `@ManyToMany(mappedBy = "feld")` |

## Zusammenfassung

### Wichtige Begriffe
- `@OneToOne` / `@OneToMany` / `@ManyToOne` / `@ManyToMany`
- `@JoinColumn` / `@JoinTable`
- `mappedBy`
- `FetchType.LAZY` / `FetchType.EAGER`
- `CascadeType`
- `orphanRemoval`
- `LazyInitializationException`
- `JOIN FETCH`

### Das sollst du können
- 1:N-Beziehungen mit `@ManyToOne` / `@OneToMany` modellieren
- 1:1-Beziehungen mit `@OneToOne` abbilden
- M:N-Beziehungen mit `@ManyToMany` und `@JoinTable` umsetzen
- Die Fetch-Strategien lazy/eager unterscheiden und anwenden
- Cascading-Optionen gezielt einsetzen
- `LazyInitializationException` vermeiden
